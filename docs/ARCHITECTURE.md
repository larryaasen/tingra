# Tingra: Architecture and Technical Plan

Tingra is a native macOS live streaming and production application, written entirely in Swift and built on Apple's modern media stack. This document describes the technical architecture and the role each component plays, including where the HaishinKit library fits in.

## Design principles

1. **Mac first, not cross platform.** Every choice favors deep integration with macOS over portability. No Qt, no OpenGL, no ported abstractions.
2. **GPU resident pipeline.** Frames stay on the GPU as `IOSurface` backed buffers from capture through composition to compression, avoiding expensive CPU round trips.
3. **Modern Apple frameworks over reinvention.** ScreenCaptureKit, AVFoundation, Metal, VideoToolbox, and AVAudioEngine do the heavy lifting.
4. **Build the distinctive parts; adopt the thankless parts.** Capture and composition are what make Tingra native and distinctive, so we own them. RTMP/SRT networking and muxing are hard, standardized, and undifferentiated, so we adopt a proven library (HaishinKit) rather than rebuild them.
5. **Plug-in first.** All core functionality ships as first party plug-ins built on the same protocol third parties use. The host, the only code that is not a plug-in, is the smallest thing that can run plug-ins and move frames.

## Vocabulary

Tingra uses Apple Pro app and broadcast terminology. GLOSSARY.md is the canonical reference for every term. The ones this document leans on hardest: **input** (anything producing frames, including **generators** that synthesize them), the **project > preset > shot > layers** hierarchy, **program** (what viewers see and hear), **compression** and **output** (the delivery components), **destination** (a streaming target), and **plug-in** / **host** / **registry** (the extensibility model; always spelled "plug-in"). Tingra does not say source, scene, egress, ingest, or encoder; see the glossary for the full list and the protocol boundary exception.

## Color and pixel format conventions

One canonical working format and one delivery standard, decided up front — mismatched or untagged color is the classic source of "slightly washed out" streams. Tingra uses the normal broadcast conventions:

- **Working format (capture → composition):** IOSurface backed `kCVPixelFormatType_32BGRA` (Metal `bgra8Unorm`), SDR, tagged BT.709. This is the "common GPU resident frame type" every input normalizes to. Normalization at the input seam is the one place pixel format and color conversion happen — YCbCr camera frames ('420v'/'420f') to RGB, wide gamut sources (Display P3 screens and cameras) down to BT.709. The program frame the compositor emits is this same format.
- **Delivery standard (compression sinks):** H.264/HEVC, 4:2:0 chroma subsampling, video (limited) range, BT.709 — with the color description written into the bitstream (VUI `colour_primaries`/`transfer_characteristics`/`matrix_coeffs` all BT.709) and matching format-description/track tags on recordings. Program resolutions stay even in both dimensions (4:2:0 requires it).
- **Every buffer is tagged.** Every `CVPixelBuffer` in the pipeline carries `CVImageBuffer` color attachments (primaries, transfer function, YCbCr matrix). Untagged buffers are a defect: untagged means the next stage guesses, and guessing is where color bugs live.
- **One conversion point.** Composition and the sinks never re-convert; they trust the tags set at input normalization. Core Image manages its own (linear) working space; hand written Metal shaders that blend or sample must be deliberate about sRGB vs. linear texture views rather than relying on defaults.
- **SDR only for now.** Wide gamut and HDR program output (P3, HLG/PQ) are out of scope until well after the app era; if they come, they enter at the same input normalization seam and become a program-wide setting, not a per-input one.

## Frame ownership across the `Input` seam

`CVPixelBuffer` and `CMSampleBuffer` are not `Sendable`, yet frames and audio buffers must cross isolation boundaries from an input's capture callback to the compositor and the sinks. Tingra resolves this with one deliberate, documented ownership rule instead of scattered ad hoc `@unchecked Sendable` (decided 2026-07-04; the rule CLAUDE.md's strict-concurrency note calls for). Every producer and consumer of media buffers observes it:

1. **Transfer at yield.** The producer (an `Input`) hands the buffer off when it yields to its `frames()`/`audio()` stream and never touches it again — no reuse, no late writes, no retained references.
2. **One holder at a time.** Exactly one consumer owns the buffer at any moment. The compositor's latest-wins slot releases its previous frame when a newer one replaces it; a sink that needs the buffer beyond its call must be handed ownership explicitly, never share it.
3. **Immutable after transfer.** Nothing writes to the buffer after the yield. Downstream stages read, composite, and encode from it only; any stage that needs modified pixels renders into a new buffer.

Under this rule a buffer is only ever read after the unique transfer point, which is exactly the guarantee `Sendable` encodes — so the wrapper types `CapturedFrame` and `CapturedAudio` in the plug-in protocol package carry `@unchecked Sendable` soundly. **Those two types are the only sanctioned `@unchecked Sendable` in the codebase.** They are the single choke point every media buffer crosses; any new `@unchecked Sendable` anywhere else needs its own documented rule and review, and "the frame path needed it" is not a reason — the frame path already has its rule here.

The rule is enforced by convention and review, not the compiler — which is why it lives in this document, is restated on the two wrapper types, and is deliberately short enough to hold in your head while writing an input.

## Engine model: host and plug-ins

The engine splits into two kinds of code.

**The host** owns the plug-in loader and lifecycle, the registries every plug-in registers into, frame transport (the GPU resident pipeline, its master clock, and the program tick that paces the compositor — see CLOCK.md), the session and state manager, the event bus and its logging sinks (see EVENTS.md), secure storage (Keychain backed, for stream keys), and authorization (Screen Recording, Camera, Microphone). The host has no features a viewer would ever see. The test for what belongs in the host: if removing it breaks plug-ins in general, it is host; if removing it breaks one capability, it is a plug-in.

**Feature plug-ins**, first party and third party alike: every capture input, every generator, every effect and transition, local recording, every streaming output, and every MCP tool surface beyond the host's own introspection tools. First party plug-ins ship inside the product and load through the same discovery path, protocol, and registries as anything a third party writes.

Each input is an implementation of an `Input` protocol; the compositor and everything downstream consume the protocol and never import a capture framework directly, mirroring the `StreamingService` seam on the output side.

```swift
// Sketch of the boundary — not final API.
protocol Input {
    var id: InputID { get }
    func start() async throws
    func frames() -> AsyncStream<CapturedFrame>   // GPU-resident CVPixelBuffers
    func stop() async
}

final class CameraInput: Input { /* AVFoundation, all Apple platforms */ }
final class DisplayInput: Input { /* ScreenCaptureKit, macOS only     */ }
final class BarsGenerator: Input { /* generator, runs everywhere      */ }
```

**Delivery and isolation on macOS:** plug-ins are separately compiled bundles. Third party plug-ins install into a known folder (e.g. `~/Library/Application Support/Tingra/Plug-ins`); first party plug-ins ship in `Contents/PlugIns/`. The shared plug-in protocol lives in its own framework built with Library Evolution for a stable module interface; the entry point is the bundle's principal class via `Bundle.principalClass`. Tingra carries the Disable Library Validation entitlement (decided; feasible since Tingra is not Mac App Store sandboxed) because third party plug-ins will not share Tingra's Team ID. Isolation policy: bundled first party plug-ins run in process; third party plug-ins are candidates for XPC isolation where the path tolerates it (outputs, automation), while capture plug-ins stay in process because the per frame IOSurface path and TCC attribution both favor it.

**Delivery on iPadOS (if an iPad build ever happens):** runtime loading of separately distributed binaries is not possible. All executable code must be embedded in the signed app bundle at build time; loading code from outside the bundle is blocked by code signing enforcement and App Store policy (guideline 2.5.2), and ExtensionKit's third party extension point hosting is macOS only. The same `Input` protocol still works, but implementations are statically linked and chosen at compile time. An iPad build would keep camera inputs and generators, lose display/window inputs (ReplayKit could become a limited replacement input later), and use pure SwiftUI in place of the AppKit half of the UI. The seam buys portability of the core, not feature parity.

**In the CLI era**, bundled plug-ins are compiled in but register through the same code path the external bundle loader will use; the loader itself can ship after the first release.

### Plug-in API stability and versioning

The plug-in protocol package is the one API Tingra can never break casually: a shipped third party plug-in must keep loading and working across Tingra updates. The policy, in force from the package's first tag:

- **SemVer, strictly, versioned independently of the product.** During the CLI era (roadmap steps 1–4, before the external bundle loader ships) the package stays at 0.x, where minor bumps may break per SemVer convention. It tags **1.0.0 when the external bundle loader ships** — the moment third parties can actually build against it. From 1.0.0 on, breaking changes land only in major versions.
- **What counts as breaking:** removing or renaming any public symbol; changing a signature or the documented semantics of existing API; adding a protocol requirement without a default implementation; tightening concurrency requirements (`Sendable`, actor isolation) on existing API. All of these wait for a major.
- **Additive evolution first.** New protocol requirements ship with default implementations so existing conformances keep compiling. When a capability changes shape, prefer a new protocol (or protocol inheritance) over mutating a shipped one.
- **Deprecate before removing.** Mark with `@available(*, deprecated, message:)` naming the replacement, keep the symbol for at least one minor release, and remove only in the next major.
- **Library Evolution stays on** (decided above) so the module interface is stable: a plug-in built against an older minor keeps loading against a newer one without recompilation.
- **Version negotiation at load, never a crash.** Each plug-in bundle declares the protocol major it was built against (an Info.plist key under `com.moonwink.tingra.*`); the host refuses an incompatible plug-in with an `error` event naming both versions and the fix (EVENTS.md), and the remaining plug-ins load normally — per the never-crash rule.
- **CI enforcement.** The protocol package's CI job runs `swift package diagnose-api-breaking-changes` against the latest release tag, so an accidental break fails the pull request instead of a third party's build.
- **The policy travels with the dependency surface.** The protocol package's only dependency is the zero dependency event bus package (EVENTS.md), which is re-exposed to every plug-in — so the same SemVer and evolution rules bind the event bus package too.

## Engine services

The engine is organized as services, each exposing its capabilities through plug-in registries: Capture (inputs, generators, input discovery, device connection and disconnection), Composition (presets, shots, layer tree, transitions, Metal renderer, effects, program/preview buses), Audio (mixer, channel strips, routing, audio effects), Compression (VideoToolbox compression sessions, rate control, local recording), Output (the `StreamingService` seam with HaishinKit backed RTMP/SRT implementations, later multiple destinations), Plug-in (discovery, lifecycle, isolation), MCP/Control (tool registry, session/state, authorization bridge), and Platform/Infrastructure (event bus, logging, secure storage, local storage, system info).

### Capture
- **ScreenCaptureKit (macOS 12.3+)** for display, window, and per application inputs. It delivers frames as `CMSampleBuffer` wrapping `IOSurface` backed `CVPixelBuffer`s, which stay on the GPU.
- **AVFoundation** (`AVCaptureSession`) for camera and microphone inputs.
- Every input normalizes to a common GPU resident frame type that the compositor consumes.
- **Two capture plug-ins, split by framework and permission** (`TingraCapturePlugIns`): `AVFoundationCapturePlugIn` contributes cameras and microphones (Camera / Microphone TCC), and `ScreenCaptureKitCapturePlugIn` contributes displays (`InputKind.display`, a pre-1.0 additive case; Screen Recording TCC) — a separate plug-in, decided 2026-07-06, because it is a different framework and a different permission, matching this services split. Display discovery reads CoreGraphics, which needs no Screen Recording authorization (like camera discovery, listing never prompts; only capturing does), and identifies each display by its `CGDisplayCreateUUIDFromDisplayID` UUID, which survives reconnection where the transient `CGDirectDisplayID` does not; capture then resolves that UUID to the current display and runs an `SCStream` delivering 32BGRA frames tagged BT.709 at the seam. Displays are not yet in the CLI's `devices` listing — they are an app-era surface (CLI.md non-goals); the CLI loads only the AVFoundation plug-in.

### Composition
- **Metal** renders the layer tree of the current shot: positioning, scaling, cropping, and layering inputs, plus transitions between shots.
- **Core Image** (Metal backed) supplies effects without hand writing every shader; drop to raw Metal shaders where custom work or performance demands it.
- The result is a single program frame per tick, still GPU resident. This buffer is the **clean seam** described below.

**The compositor (roadmap step 6).** Composition lives in its own package, **`TingraComposition`** — a host-side engine library, a sibling of `TingraHost` depending only on the plug-in protocol package and the event bus (decided 2026-07-06; it is not a plug-in — effects and transitions plug *into* it — but it is a large, distinct concern that would bloat the minimal host, and the protocol-only dependency keeps it testable in isolation with a synthetic clock and a mock renderer). The `Compositor` is the tick-paced engine: it holds a latest-wins slot per input (a fill task drains each input's `frames()` into it) and the current `Shot`, and on each program tick it snapshots the shot and slots, renders the layer tree, and yields the composited program frame stamped with the tick's clock time. This is the step-6 realization of the model the single-input `ProgramPacer` stood in for during the CLI era — **same tick, same latest-wins slot semantics, same timestamps, with "take the latest frame" replaced by "render the layer tree"** across every input (CLOCK.md, "The tick before composition exists"). A stalled input keeps its last frame in its slot; the program is a live canvas from the first tick, showing the shot's background before any input delivers. The CLI's single-input `StreamSession` keeps using `ProgramPacer` for now — wiring the compositor into the stream path is a step-7 concern (multi-input streaming), so step 6 leaves the shipped CLI path untouched.

- **The layer tree** is plain values: a `Shot` is an ordered array of `Layer`s (bottom to top) over a `BackgroundColor`; a `Layer` names its input by `InputID` (not the input itself, so a shot is comparable and switchable live) and carries a normalized, top-left-origin destination `frame` and an `opacity`. The compositor resolves each id to that input's latest frame at tick time.
- **The renderer sits behind an internal `ShotRenderer` seam** whose default, `CoreImageShotRenderer`, composites with a Metal-backed `CIContext` (Core Image supplies the compositing without hand-writing shaders; raw Metal shaders wait for the effects/transitions step where custom work demands them). The renderer is created and used entirely inside the compositor's tick task (like the generator plug-in's `BarsRenderer`), so it needs no `Sendable`/`@unchecked Sendable`; a software `CIContext` makes placement, the top-left→bottom-left Y-flip, opacity, and BT.709 tagging deterministically unit-testable with no GPU.

**Presets and shots (roadmap step 7).** The first slice of the production feature set lands as data model plus engine plumbing (the layer-tree editor and the audio mixer follow in later step-7 iterations). A **`Preset`** is a named, persisted collection of `Shot`s — the `project > preset > shot > layers` hierarchy from GLOSSARY.md, one level up from the shot. `Preset`, `Shot`, and `Layer` are all plain `Codable` value types: the serialized form is the **project / scripting contract** (CLAUDE.md, "Data Models"), so keys are stable camelCase, a shot carries a stable `id` (`ShotID`) and a user-facing `name`, and a `Layer`'s `frame` is flattened to explicit `x`/`y`/`width`/`height` rather than the nested arrays a raw `CGRect` would synthesize. Three decisions, recorded here (2026-07-06):
  - **The active shot is session state, not part of the persisted preset.** A `Preset` holds only its shots (audio configuration, connected inputs, and destinations join it in later iterations); *which* shot is currently on program is live session state (GLOSSARY.md, "Session") owned by the `Compositor`, not a field of the saved document. This keeps the project file a pure description and matches "the session survives across tool calls" — it is not something you save and reopen.
  - **Switching a shot is driven by `take(shotID:transition:)`, defaulting to a cut.** The compositor gains `loadPreset(_:)` (load a preset's shots, cut to the first) and `take(shotID:transition:)` (take the named shot to program starting the next tick, with the given transition — a cut by default). `setShot(_:)` remains the low-level "render exactly this shot" path (used by the pre-preset `ProgramPacer`-era callers and tests), always a hard cut, and bypasses the preset's active-shot tracking. Taking an id absent from the loaded preset is recoverable — it reports a `program.take` error event and leaves the program unchanged, never a crash. Both `preset.loaded` and `program.take` are control-plane events on the bus, never per-frame traffic.
  - **The app builds shots with fixed ids so identity survives a rebuild.** `apps/tingra`'s built-in shots (picture-in-picture, display, camera) use fixed `ShotID` tokens rather than fresh UUIDs, so re-deriving the preset when the input selection changes preserves the shot's identity — the operator stays on the shot they had taken rather than snapping back to a default. User-authored shots get fresh UUIDs. *(Partially superseded: since the project-file iteration a selection change rebinds rather than rebuilds, so the fixed tokens now only identify the seed of a fresh project and key the switcher's per-shot tap names; since the shot-management iteration the seeded shots are just shots — renameable and removable like any other. See "Project save/load" and "Shot management".)*

**Transitions: cut and dissolve (roadmap step 7, second iteration).** A **`Transition`** (GLOSSARY.md: cut, dissolve, wipe, custom shader) lands as a plain `Codable` value type in `TingraComposition` — the same project-file contract as `Preset`/`Shot`/`Layer` (stable camelCase `kind`/`durationSeconds` keys), so it round-trips exactly wherever it is later persisted. This iteration implements only `cut` and `dissolve(duration:)`; `wipe` (since landed — see "Wipe transitions") and custom shader based transitions (since landed — see "Custom-shader transitions") remain unrepresented cases for later iterations. Four decisions, recorded here (2026-07-08):
  - **A transition is passed to `take(shotID:transition:)`, not yet a field on `Shot` or `Preset`.** Nothing about *how* a shot is taken belongs to the shot itself (the same shot might be cut to in a rehearsal and dissolved to on air); a per-preset or per-shot default transition is a plausible later addition, but this iteration keeps the caller in control and the data model unchanged. *(Since landed per-shot — the caller-in-control contract intact — see "Per-shot default transitions".)*
  - **A dissolve's duration is counted in ticks, not wall-clock time.** `take` converts `duration` (seconds) to a whole number of program ticks (`round(duration × frameRate)`, minimum 1) at call time, and the `Compositor` counts ticks elapsed rather than comparing clock timestamps. This keeps the transition's progress exact and deterministic under both the master clock and a synthetic test clock (CLOCK.md, "The program tick" — nothing outside the tick stream should decide how much time has passed), and guarantees a dissolve always completes (a zero or negative duration still finishes on its first tick) rather than stalling.
  - **The crossfade is a `ShotRenderer` seam, not compositor-side pixel work.** `ShotRenderer` grew `renderDissolve(from:to:progress:frames:format:time:)` alongside `render(shot:frames:format:time:)`; `CoreImageShotRenderer` renders both shots' layer trees with its existing per-shot compositing path, then alpha-blends the incoming image over the outgoing one at `progress` — the same "fade toward what's underneath" math the layer-opacity path already used, just applied to a whole shot instead of one layer. The compositor stays pixel-agnostic: it snapshots which tick of the dissolve is current and hands the renderer two shots and a progress value.
  - **The active shot updates immediately, matching the cut's contract.** `take(shotID:transition:)` sets `activeShotID` (and the switcher's highlighted shot) to the incoming shot right away, whether the transition is a cut or a dissolve in progress — a caller reads "what's *going* to program," not "what's fully visible this instant." The compositor keeps rendering the crossfade tick by tick until it completes, then settles on a plain render of the incoming shot.

**The layer-tree editor (roadmap step 7, third iteration).** The operator edits the layer tree of the shot currently selected in the switcher — add a layer bound to an input, remove a layer, reorder the bottom-to-top stack, and adjust a layer's normalized top-left-origin frame and its opacity — live on program. The engine surface is one new `Compositor` method; the editing UI (`LayerTreeEditorView`) and the pure edit operations (`LayerTreeEdit`) live in `apps/tingra`. `Layer` is unchanged — still a plain `Codable` value type, the same project/scripting contract. Per-layer effect/filter chains are a later "Effect" iteration (GLOSSARY.md, "Effect"). Five decisions, recorded here (2026-07-11):
  - **Live edits flow through `Compositor.updateShot(_:)`, which replaces the loaded preset's shot in place.** It matches the edited shot to the loaded preset by `ShotID` and swaps it in; when that shot is on program, the very next tick renders the edited layer tree — no separate "apply" step, because the program is a live canvas at the tick rate (CLOCK.md, "The program tick"). While a dissolve is in progress *toward* the edited shot, the dissolve continues toward the edited tree (the outgoing side of a dissolve is not retro-edited — it is on its way off program). `updateShot` never changes `activeShotID`: editing a shot is not taking it. An id absent from the loaded preset is recoverable — a `shot.update` error event, the preset untouched, never a crash.
  - **A successful `updateShot` reports no event.** A live editor drives it at gesture rate — a slider drag calls it many times a second — so a per-update control-plane event would flood the bus (EVENTS.md admits control-plane traffic only, and this is closer to per-frame). User-action observability comes from the app's `tap` events instead, reported once per gesture (a slider reports its final value when the drag ends). If a scripting surface (an MCP tool) later edits shots, that tool reports its own control-plane event.
  - **Edits persist in the in-memory preset — session-scoped, not yet saved.** The compositor's loaded shot pool holds the edited shot, so the edit survives `take` switches away and back within the session; the app's session preset mirrors it. Project-file save/load is a later iteration. In `apps/tingra`, changing the camera/display **selection** still re-derives the built-in shots from `ProgramLayout` and discards layer edits — the built-in preset is a pure function of the selection; user-authored presets that own their layer trees arrive with the project file. *(Superseded 2026-07-12 by the project-file iteration: edits now autosave to the project document, and a selection change rebinds rather than rebuilds — see "Project save/load".)*
  - **A new layer binds to any discovered camera or display, and lands on top of the stack, full-frame, opaque** — you add a layer to see it. The app starts an input the first time a layer references it and stops it when no shot references it and it is not the selected camera/display (the same reconfigure pass that already manages the selection; an input that cannot start is reported and its layer simply contributes nothing). Generators stay out of the add-layer choices for now: `InputKind.generator` cannot say whether a generator produces video or audio, and a video/audio capability on the `Input` seam is a deliberate later protocol addition, not something to bolt on in a UI iteration.
  - **Layers are addressed positionally; the editor lists them topmost first.** `Layer` deliberately gains no id — it stays a plain value, and per GLOSSARY.md layers stack in a defined order, so the bottom-to-top index *is* the identity. The editor displays the stack topmost-first (the design-tool convention, so "up" means "in front") while every operation addresses the underlying bottom-to-top array index; an out-of-range index (a stale editor selection) returns the shot unchanged.

**Project save/load (roadmap step 7, fourth iteration).** The preset the operator already has — including layer-tree edits — persists across launches as a **project** file (GLOSSARY.md: the saved document for a whole show). The document type, **`Project`**, lands in `TingraComposition` beside `Preset`/`Shot`/`Layer`; the file handling (`ProjectStore`) and the save/load policy live in `apps/tingra`. Shot management UI (landed in the next iteration — see "Shot management") and multiple presets in the UI (landed 2026-07-14 — see "Multiple presets in the UI") were later iterations, though the document format holds a preset array from the start. Five decisions, recorded here (2026-07-12):
  - **`Project` is a versioned, plain `Codable` value type; v1 of the document holds the presets only.** The same project/scripting contract as `Preset`/`Shot`/`Layer`: stable camelCase keys (`version`, `presets`), exact round-trip, forgiving decode where fields are optional (`presets` defaults to empty). `version` is **required** — a document must declare its format so future versions can migrate it — and decoding a document *newer* than the build understands throws with a developer-facing message rather than silently loading a document whose unknown fields the next save would clobber. Destination configurations and settings join the document (with a version bump) in later iterations. *(Amended 2026-07-18: pre-release the format stays version 1 and grows within it — bumps and migrations begin with the first format change after the first release; see "Per-shot default transitions".)*
  - **The app autosaves one project file; explicit Save/Open commands wait for the document UI.** The document lives at `~/Library/Application Support/Tingra/Default.tingraproject` (the established Tingra home, beside the daemon's socket) — pretty-printed, sorted-key JSON written atomically. Saves are **debounced**: each edit restarts a one-second delay, so a slider drag's per-gesture edits coalesce into a single write (the same flood reasoning that keeps successful `updateShot` calls off the event bus), with an immediate save when a fresh project is seeded and a flush on engine stop. Autosave fits the live-canvas model — edits are already live on program, so an "unsaved changes" state would be a fiction; a File menu (Save/Open/Save As, multiple projects, a user-chosen location) arrives with the document-based UI. Saves and loads are control-plane events (`project.loaded`/`project.seeded`/`project.saved`, errors as `project.load`/`project.save`); the active shot and the dissolve toggle remain session state, never part of the document.
  - **The built-in `ProgramLayout` arrangement seeds a fresh project only.** At launch the app loads the document's first preset as the session preset (preserving any further presets in the array verbatim across saves). Only when there is no file — or the file holds no presets — does `ProgramLayout` derive the picture-in-picture/display/camera shots from the initial selection, once, and save the result; from then on the project owns its shots.
  - **A selection change rebinds, never rebuilds.** With shots persisted, the camera/display pickers stop re-deriving the built-in shots (which discarded edits — the caveat recorded under "The layer-tree editor"): a picker change **recasts which device plays that role**, rebinding every layer bound to the previously cast device to the new choice across all the preset's shots, keeping each layer's frame and opacity. Picking "None" parks the role's device — the input stops, its layers keep their binding and simply contribute nothing (the existing disconnected-input semantic) — and picking a device again rebinds them to it. A layer bound to an input that is no longer discovered stays bound and dormant until the device returns; recasting it onto a different device is a layer-tree editor operation (remove the stale layer, add one bound to the new device), since an undiscovered id's kind is unknown and the pickers only cast among discovered devices.
  - **An unreadable project file is set aside, never overwritten.** A file that exists but does not decode (corrupt, or written by a newer Tingra) is reported as a `project.load` error event and renamed to a `.unreadable` sibling before a fresh project is seeded — the app never silently destroys the operator's document, and never crashes over one.

**Shot management (roadmap step 7, fifth iteration).** The operator manages the shots of the preset they already have, from the switcher: add a new shot, duplicate a shot, rename a shot, and remove a shot — every edit autosaved through the existing project-document path. `Preset`, `Shot`, and `Layer` are unchanged (the same plain `Codable` project/scripting contract, no document version bump); the engine surface is two new `Compositor` methods beside `updateShot(_:)`; the pure shot operations (`ShotEdit`) and the switcher UI live in `apps/tingra`. Multiple presets in the UI (landed — see "Multiple presets in the UI"), shot reordering in the switcher, and per-shot default transitions (since landed — see "Per-shot default transitions") were explicitly deferred to later iterations. Five decisions, recorded here (2026-07-12):
  - **Adding and removing reach the loaded preset through granular `Compositor` methods — `addShot(_:at:)` and `removeShot(shotID:)` — not a reload path.** They extend the in-place editing surface `updateShot(_:)` established (a rename, and edits to a duplicate's source, already flow through it), so the active shot and any in-progress dissolve survive by construction — a reload path would have had to re-specify how both are preserved. `addShot` inserts at the given switcher position (default: the end) and **never changes the program** — adding a shot is not taking it, the same contract as `updateShot`'s "editing a shot is not taking it"; a duplicate id is a recoverable `shot.add` error event, the pool untouched. Unlike `updateShot`, both report success events (`shot.added`/`shot.removed`): they are discrete button-driven actions, not gesture-rate edits, so the control-plane traffic is right (EVENTS.md).
  - **Removing the shot on program cuts to the adjacent shot — never a dead program.** The shot now occupying the removed shot's switcher position (its follower), or the new last shot when the removed one was last; the cut clears any in-progress dissolve toward the removed shot. Removing the last remaining shot leaves the pool empty and the program on the background-only canvas with no active shot — still a live canvas at the tick rate (the compositor's step-6 contract), not a dead program. Removing a shot that is only the *outgoing* side of an in-progress dissolve lets the dissolve finish from its snapshot — the outgoing shot is on its way off program, the same rule `updateShot` recorded.
  - **The UI is an explicit Add Shot button on the switcher row, plus per-shot commands in each shot button's context menu** (Duplicate, Rename…, Remove Shot). Adding is not scoped to any one shot, so it is a visible button — and it stays available when the preset has no shots, so the operator is never stranded on an empty switcher; the per-shot commands ride the macOS right-click convention, keeping the shot buttons' single click reserved for the on-air take. Rename is an alert with a text field. Remove is immediate (a destructive-role menu item, no confirmation dialog): shots are "quick to create, switch, and discard" (GLOSSARY.md, "Shot"), and undo arrives with the document-based UI.
  - **A new shot starts empty over opaque black.** The layer-tree editor adds layers — an empty shot renders the background, so the operator composes on the live canvas rather than inheriting an arrangement they must undo. A duplicate copies the source's layer tree and background under a "<name> copy" name. A rename to an empty (or whitespace-only) name is ignored: a switcher button needs a label, so `Shot.name`'s empty-for-ad-hoc case stays an API affordance, never a UI outcome.
  - **User-authored shots get fresh UUIDs; the seeded fixed-id shots are just shots now.** The fixed `pip`/`display`/`camera` tokens remain what `ProgramLayout` seeds a fresh project with (and what keys the switcher's per-shot tap names), but nothing else distinguishes them — they rename and remove like any other shot. Their original rationale, identity surviving a selection-change rebuild, was already superseded by rebind-never-rebuilds ("Project save/load").

**Streaming the program (roadmap step 7, sixth iteration).** The operator puts the program they already have on air: a destination (RTMP/RTMPS URL + stream key) and a Start/Stop control with live status, in `apps/tingra`. This is the app reaching the output seam for the first time — the compositor's program frames flow to the HaishinKit-backed `StreamingService` through the same `OutputRegistry` the CLI resolves against (`apps/tingra` gains the `TingraOutputPlugIns` dependency and activates `HaishinKitOutputPlugIn` alongside the capture/generator plug-ins). Scoped to streaming the composited program with microphone pass-through audio; the audio **mixer** and **multiple presets** in the UI landed in later iterations (see "The audio mixer", "Multiple presets in the UI"), while **wipe** transitions (since landed — see "Wipe transitions"), custom-shader transitions (since landed — see "Custom-shader transitions"), shot **reordering** (since landed — see "Shot and preset reordering"), and **SRT / multiple destinations** (roadmap step 8) remain deferred. Six decisions, recorded here (2026-07-12):
  - **`StreamSession` learns an external program source rather than the app driving a `StreamingService` directly.** The host's `StreamSession` already owns the shared `T0` timeline, the reconnect attempts, the 10-second stability window, the periodic stats, the duration timer, and the pass-through audio path — the whole live-stream lifecycle the CLI proved. Rebuilding that in the app would duplicate and diverge from it. So `StreamSession` gained a `VideoSource` enum: `.input(any Input)` is the CLI's single-camera path (the session paces it through `ProgramPacer` and owns its device lifecycle), and `.program(AsyncStream<CapturedFrame>)` is the app's path — the compositor's already tick-paced program frames, consumed as-is with no second pacing, with the compositor and its inputs owned by the caller. Both stamp frames on the master clock, so the identical `T0` rebase applies; the reconnect/stability/stats machinery is untouched. The existing `init(videoInput:…)` is preserved (it delegates to `.input`), so the CLI and the daemon's `StreamCoordinator` are unchanged.
  - **The app tees its one program drain; it never opens a second `Compositor.programFrames()` consumer.** `Compositor.programFrames()` is single-consumer (a new call finishes the previous), and the preview already drains it. So the app's one program drain forwards each composited frame to both the preview relay and — while a stream is live — a fresh `AsyncStream` feeding the session, preserving the compositor's single-consumer contract. Both readers only read the immutable program frame (the frame-ownership rule), exactly as `StreamSession` already feeds one frame to both the stream and recording sinks.
  - **The stream key lives only in the host's Keychain-backed secure storage.** A new host component — `SecureStorage` (a seam) with a `KeychainSecureStorage` implementation (the data-protection keychain, keyed by the destination URL under Tingra's identifier namespace) — holds the key. It never enters the project document, an event param, or a log (CLAUDE.md, "Error Handling"; EVENTS.md, Redaction). The app's key field is view-local `@State`, handed to `startStreaming(streamKey:)` only, which stores it and builds the seam's `Destination` (which is deliberately not `Codable`); the field is prefilled by reading secure storage for the current URL. A seam so the app runs against the real Keychain and tests against an in-memory double (no unlocked keychain, no prompt on CI).
  - **The destination URL joins the project document; the key does not.** Anticipated by the save/load iteration ("destinations join with a version bump — keys excluded"), the `Project` document goes to **version 2**, adding an optional, key-free `ProjectDestination` (URL only) beside the presets. An older v1 document decodes forward with the destination nil; a v2 document read by a v1 build throws (the newer-version rule). One destination in v1 of the app (the one-active-session rule); multiple destinations are step 8. The URL persists through the same debounced autosave as the layer-tree edits; the key is fetched from secure storage under that URL. *(Superseded 2026-07-18 by the pre-release versioning rule: the v2 bump is retired — the format is version 1 with the destination simply part of it; see "Per-shot default transitions".)*
  - **The streaming controls live in `ContentView`, driven by the bus's status events — never polled.** A streaming panel below the input pickers holds the URL field, the key `SecureField`, a microphone picker (discovered mics, or "No audio"), a status label, and the Start/Stop button (every button reports its `tap` first). `EngineModel` observes the event bus for the `stream.started`/`stream.stats`/`stream.reconnecting`/`stream.reconnected`/`stream.stopped` events `StreamSession` already emits and maps them to a `StreamStatus` the panel renders (idle, connecting, live, reconnecting, stopped, error), with the latest `stream.stats` bitrate/fps shown beside the Live label — the same event-driven model the daemon's status sink uses, no poll loop.
  - **Audio is microphone pass-through on the shared timeline — the CLI's model — and reconnect defaults match the CLI.** The selected microphone is a `StreamSession` audio input (session-owned lifecycle, separate from the compositor's video inputs — the compositor has no audio stage yet); the mixer landed in the next iteration (see "The audio mixer"). The app uses `StreamSession.Policy`'s defaults (3 attempts, 2-second delay, 10-second stability window, stats every 5 seconds), session policy rather than UI knobs, until someone needs to tune them.

**The audio mixer (roadmap step 7, seventh iteration).** The last unbuilt engine service: the **mixer**, the audio surface of the engine, combining every audio input into the program mix through one **channel strip** per input (GLOSSARY.md) — replacing the single-microphone pass-through the app streamed with. Scoped to level and mute per strip; **pan (since landed — see "Per-strip pan"), routing (since landed — see "Per-strip routing"), and audio effect chains** (the rest of the GLOSSARY.md strip) and **monitoring/meters (meters since landed — see "Per-strip meters")** are explicitly deferred to later iterations, as are wipe transitions (since landed — see "Wipe transitions"), custom-shader transitions (since landed — see "Custom-shader transitions"), multiple presets in the UI (since landed — see "Multiple presets in the UI"), shot reordering (since landed — see "Shot and preset reordering"), and SRT/multiple destinations (step 8). Six decisions, recorded here (2026-07-12):
  - **The mixer is its own engine package, `packages/TingraAudio` — a sibling of `TingraComposition`, not part of it and not host.** Audio is a distinct engine service from Composition in this document's own service list (mixer/strips/routing/effects vs. shots/layers/renderer), and `TingraComposition`'s charter is deliberately the *video* program — folding the mixer in would blur both. The same host/plug-in boundary reasoning that carved out `TingraComposition` applies verbatim: removing the mixer breaks one capability (program audio), not plug-ins in general, so it is not host; audio effects and taps plug into it later, so it is not a plug-in — a host-side engine library depending only on `TingraPlugInKit` + `TingraEventBus`, testable with a synthetic clock and scripted inputs, no hardware, no TCC.
  - **A clock-paced software mix loop, not an `AVAudioEngine` graph.** `AudioMixer` runs a **mix tick** on the injected `EngineClock` (1024-frame blocks at 48 kHz ≈ every 21.3 ms), summing every unmuted strip's queued samples scaled by its level — so mixing is exactly as deterministic under the synthetic test clock as the compositor's tick, and strips attach and detach dynamically with no audio-graph rebuild. AVFoundation is used only for format normalization: a persistent per-channel `AVAudioConverter` (task-confined, like the compositor's renderer) converts each strip's incoming audio **once, at channel intake** — float32, deinterleaved, at the mix rate — the audio mirror of "conversion happens once, at input normalization". Mono strips spread equally into both program channels, stereo maps left/right, channels past two are dropped; each channel queue is a FIFO capped at one second (oldest samples drop at the cap, bounding a drifting-fast input's latency; an underrunning or stalled strip contributes silence and never stalls the mix — muted strips keep draining so unmute never replays stale audio). The mix is a float sum with no bus clamp: float32 has headroom, and delivery saturates at compression. An `AVAudioEngine` manual-rendering graph remains a known alternative if measured quality ever demands Apple's resampler across the whole path.
  - **Mixed blocks are stamped with the mix tick's master clock time — the audio analog of the program video frame's PTS rule.** A mixed block spans multiple inputs, so no single capture time exists for it; and because tick deadlines are absolute positions (`T0 + n × blockDuration`, CLOCK.md), consecutive blocks form a perfectly contiguous, monotonic PTS sequence into the sinks. The capture-time rule is untouched where it applies: each strip's *incoming* buffers still carry true `AVAudioTime` host-time PTS, and the CLI's single-microphone pass-through still delivers them unmodified. Recorded in CLOCK.md's timestamp table alongside the program video rule.
  - **`StreamSession` gained an `AudioSource` enum mirroring `VideoSource` — the streaming iteration's design, reused.** `.input(any Input)` is the CLI's pass-through microphone (session-owned lifecycle, capture-cadence delivery — byte-for-byte the previous behavior; `init(videoInput:audioInput:…)` is preserved, so the CLI and the daemon's `StreamCoordinator` are unchanged), and `.program(AsyncStream<CapturedAudio>)` is the mixer's already mix-tick-paced program audio, consumed as-is with the mixer and its inputs owned by the caller. Both stamp media on the master clock, so the identical `T0` rebase applies. A program-audio session reports the stable `"mix"`/`"Mix"` identity in `stream.started` (the audio counterpart of the video source's `"program"`), and the app now always streams with program audio — an all-muted mixer streams silence, the way an empty shot streams the background canvas.
  - **The app's strip policy: one strip per discovered audio input, the first unmuted at unity, and mute stops the device.** The seeded default is the successor of the CLI-era "first microphone streams"; every other strip starts muted. Muting a strip both silences its channel and **stops its device** — the macOS microphone indicator stays honest, a muted microphone is not captured — and unmuting starts it again (the same reconfigure-pass pattern, coalesced, that manages the video inputs; an input that cannot start is reported and its strip simply contributes silence). The engine-side mute stays independent of device lifecycle — that pairing is app policy, and a future monitoring path may keep muted devices running. The mixer runs from boot like the compositor (a live canvas of silence until a strip delivers), and the app's one program-audio drain tees mixed blocks into the live session exactly as the one program-frame drain does — no second `programAudio()` consumer. Level changes are gesture-rate and report no engine events (the `updateShot` rule); observability is the panel's `tap` events plus the discrete `mixer.started`/`mixer.stopped`/`mixer.channels` control-plane events, all in the `audio` domain.
  - **Strip settings are session state this iteration, like the active shot — not yet part of the project document.** No document version bump: today's strip list is *derived from discovery* (one strip per discovered input), not an authored set, and persisting device-keyed levels before **routing** defines what an authored channel is would bake in a shape that iteration would immediately churn. Levels and mutes join the persisted **preset** (where "audio configuration" was always slated to live — see "Presets and shots") when routing lands. *(The "version bump that iteration will need" no longer applies: pre-release the format grows within v1 — see "Per-shot default transitions". Since landed: strip settings persist as the preset's authored `audioChannels` — see "Per-strip routing".)*

**Multiple presets in the UI (roadmap step 7, eighth iteration).** The operator switches among — and manages — the **project's** presets, not just the document's first: a preset switcher row above the shot switcher, mirroring the shot-management UI one level up the `project > preset > shot` hierarchy (a row of preset buttons, an Add Preset button, per-preset context-menu commands: Duplicate, Rename… via the same alert pattern, Remove Preset). The document format already held a preset array from the start ("Project save/load"), so `Project`/`Preset`/`Shot` are unchanged — **no document version bump, the format stays v2**; the engine surface is one changed `Compositor` contract plus one read accessor; the pure preset operations (`PresetEdit`) and the switcher UI live in `apps/tingra`. Shot reordering (since landed — see "Shot and preset reordering"), wipe transitions (since landed — see "Wipe transitions"), and SRT/multiple destinations (step 8) remain deferred. Five decisions, recorded here (2026-07-14):
  - **The active preset is session state, like the active shot — the document gains no "which preset is active" field.** The project file stays a pure description of the show; *where the operator is standing* in it is live session state (GLOSSARY.md, "Session"), exactly the reasoning that kept the active shot out of the preset. At launch the app adopts the document's **first** preset, the established load rule; the presets save in switcher order, never reordered to record activity.
  - **Switching presets never interrupts what is already playing out** — GLOSSARY.md's "Preset" entry already promised this ("switching presets is seamless"), and `Compositor.loadPreset(_:)`'s contract now delivers it. Three cases: when the on-program shot's id exists in the incoming preset, it **stays on program, adopting the incoming preset's version** of the shot (the `updateShot` rule applied across a preset switch; an in-progress dissolve continues toward the adopted tree); when it does not, the outgoing shot keeps rendering as a **held snapshot** — `activeShotID` goes nil, no switcher button highlights, and the new `programShot` accessor names what renders — until the operator takes a shot from the new pool (a dissolve completes toward the snapshot); only when *nothing* from a preset is on program (the first load after boot, or after a direct `setShot`) does loading cut to the first shot, the previous behavior, now the boot case. The app keeps a held snapshot's inputs running by including `programShot`'s layers in the reconfigure pass's referenced set, and reconfigures after the take that finally replaces the snapshot so its no-longer-referenced inputs stop (a dissolve *from* the snapshot may lose the snapshot's device feed mid-blend if nothing else references it — accepted, the same class of edge as unplugging a dissolve's outgoing device). `preset.loaded` gained an `activeShot` param reporting the outcome (the held case reports `"held"`).
  - **Preset management is app-level state over the same one-loaded-preset compositor — no new `Compositor` pool-of-presets surface.** The compositor's charter stays "render the loaded preset"; the *project's* preset list is document state, owned by `EngineModel` (its `presets` array, with the live session `shots` synced back into the active preset's slot before every save, switch, and duplicate) with the pure operations in `PresetEdit`, `ShotEdit`'s design one level up. Adding is **not** switching (the "adding a shot is not taking it" contract, one level up): a new preset appends empty — no shots, the "new shot starts empty" reasoning — and the operator switches to it to author it. A **duplicate copies the source's shots verbatim, shot ids included**, under a fresh `PresetID` and a "<name> copy" name: preserving shot ids is what makes a copy a safe place to rework a live preset, since switching between original and copy holds the on-program shot by id match. Renames trim and ignore empty names (the switcher-button rule). Add, duplicate, rename, and remove report app-level control-plane events (`preset.added`/`preset.renamed`/`preset.removed` — discrete actions, the `shot.added` reasoning), and every switch reports the compositor's `preset.loaded`; all edits autosave through the existing debounced project-document path, while a switch alone saves nothing (session state).
  - **Removing the active preset switches to the adjacent preset and takes its first shot — unless an id match holds the program; the last remaining preset cannot be removed.** Removal follows the shot-removal rule one level up: what the operator deletes leaves the air, so if no shot id in the adjacent preset matches the on-program shot, the app cuts to that preset's first shot (or the background-only canvas when it has none) — never a dead program, and never a snapshot of a deleted preset lingering on air. A project always holds at least one preset (the Remove command disables on the last one): unlike shots, there is no live-canvas fallback one level up — the whole switcher/editor surface operates *within* a preset, and an empty preset array would just make the next launch re-seed the built-in arrangement over the operator's intent.
  - **Audio configuration stays session state — explicitly deferred again, no v3 bump in this iteration.** The mixer iteration's calculus is unchanged by multiple presets: the strip list is still *derived from discovery*, not authored, so per-preset levels/mutes before **routing** defines an authored channel would persist a shape routing will churn. The mixer's strips therefore carry across a preset switch untouched (the program mix is not interrupted — consistent with the video side's seamless rule), and "audio configuration joins the persisted preset" remains pinned to the routing iteration ("The audio mixer", "Presets and shots"). *(Since landed — see "Per-strip routing": per-preset audio configuration arrived exactly there, with "strips carry across untouched" surviving as the no-authored-audio case.)*

**Shot and preset reordering (roadmap step 7, ninth iteration).** The operator changes the switcher order of the shots they already have — completing the shot-management story (add/duplicate/rename/remove landed in "Shot management") — and, at the same stroke and for the same reason, the order of the project's presets one level up. Switcher order *is* the persisted array order (the shots of a `Preset`, the presets of a `Project`), so **no document version bump — `Project`/`Preset`/`Shot` are unchanged, the format stays v2**; the engine surface is one new `Compositor` method beside `addShot(_:at:)`/`removeShot(shotID:)`, and the reordering commands live in `apps/tingra`'s existing switcher context menus. Wipe transitions (since landed — see "Wipe transitions"), custom-shader transitions (since landed — see "Custom-shader transitions"), and SRT/multiple destinations (step 8) remain deferred. Four decisions, recorded here (2026-07-14):
  - **Reordering shots reaches the loaded preset through a granular `Compositor.moveShot(shotID:to:)` — not a reload path — so the program survives by construction.** It extends the in-place pool-edit surface `addShot`/`removeShot`/`updateShot` established: `moveShot` reorders `state.shots` and touches **nothing else** — `activeShotID`, the shot the tick renders, and any in-progress dissolve are all left exactly as they were, so the on-program shot keeps its identity and simply sits at a different index. **Reordering is not taking: the program never changes** (unlike `removeShot`, which may cut, and like `updateShot`/`addShot`, which never do). What a reorder *does* change is which shot a future `removeShot` cuts to — its adjacency rule reads the switcher order — which is exactly the point of ordering. The destination index is index-based and clamped, matching `addShot(_:at:)`'s index-based insertion (the general pool primitive; the UI owns the "left/right" concept and passes `currentIndex ± 1`); an unknown id is a recoverable `shot.move` error event, a no-op move (same position) reports nothing.
  - **The UI is Move Left / Move Right context-menu commands — not drag-and-drop.** The shot buttons' single click is reserved for the live on-air take (`take`), an action with immediate program consequences; layering a drag gesture onto that same control would make a mis-started take a reorder or vice-versa on a *live* switcher — the strongest argument against drag here. The commands instead join the per-shot context menu beside Duplicate / Rename… / Remove, where the other shot-management commands already live (the macOS right-click convention for per-item actions), and mirror the layer editor's **Move Up / Move Down** exactly one level up — rotated to the switcher's horizontal axis (Left = earlier, Right = later), disabled at the ends the same way. Drag-to-reorder remains a defensible later addition for a non-live authoring surface; the take-reserved live switcher is not it.
  - **A discrete reorder reports a `shot.moved`/`preset.moved` control-plane event — the `shot.added`/`shot.removed` reasoning, and the reason the mechanism decision matters.** A menu command moves one step at a time — a discrete, button-driven action, not a gesture-rate stream — so a per-move event cannot flood the bus (EVENTS.md), exactly the line that put events on add/remove but kept them off the slider-driven `updateShot`. Had the mechanism been drag-and-drop (a continuous gesture firing many reorders mid-drag), the answer would have flipped to tap-only with the drag-end `tap` carrying the observability, the `updateShot`/level-slider rule — so the two decisions are linked, as the mechanism choice foretold. The event carries the moved item's id and name plus the `from`/`to` indices.
  - **Preset reordering ships in the same iteration — the same operation one level up, app-level only.** Symmetry earns it: the multiple-presets iteration deliberately mirrored shot management (presets got add/duplicate/rename/remove to match shots), and reordering only shots would reintroduce the asymmetry that iteration removed. It is also nearly free and genuinely useful: presets are the *project's*, owned by `EngineModel` (not the compositor, which holds only the one loaded preset), so `movePreset(_:to:)` is a pure document-state array move with **no `Compositor` surface, no `loadPreset`, no reconfigure** — even simpler than the shot path. Order is meaningful because the app adopts the **first** preset at launch ("Multiple presets in the UI"), so promoting a preset to the front sets the next session's default; the live session `shots` sync back into the active preset first (the established pre-edit step), and the reorder reports `preset.moved` and autosaves through the existing debounced project-document path. The active preset (session state) is unaffected by a reorder — `activePresetID` is untouched, so reordering never restarts the program.

**Wipe transitions (roadmap step 7, tenth iteration).** The third transition kind (GLOSSARY.md, "Transition": a **wipe** is a directional reveal), completing the transitions story cut and dissolve opened ("Transitions: cut and dissolve"): the incoming shot appears across the frame behind a boundary sweeping from a frame edge to the opposite edge over the transition's duration. The tick-counted timing spine is untouched — a wipe is the dissolve's `PendingTransition` with a different renderer path — so the new work is one `Transition` case, one `ShotRenderer` requirement, one Core Image implementation, and the switcher UI. Custom shader based transitions (since landed — see "Custom-shader transitions") and SRT/multiple destinations (step 8) remain deferred. Five decisions, recorded here (2026-07-18):
  - **`Transition` gains `.wipe(edge:duration:)` with a four-edge `WipeEdge` (`left`/`right`/`top`/`bottom`) — an additive change to the project/scripting contract.** The Codable form stays stable camelCase and round-trip exact: `kind: "wipe"` plus an `edge` key (the `WipeEdge` raw value) beside the existing `durationSeconds`; both are optional on decode (left edge, half-second default — the dissolve's forgiving-decode rule), an unknown `edge` throws, and `cut`/`dissolve` encode byte-for-byte as before. Four straight edges are the whole directional-reveal vocabulary of a broadcast switcher's basic wipe; a smaller set (horizontal only) would bake in an arbitrary limitation, while diagonals, irises, and patterns are shapes, not edges — they belong to custom shader based transitions, a later iteration, not extra enum cases here. Edges are named in the operator's top-left-origin screen terms, matching `Layer.frame`.
  - **The seam grows a per-kind `renderWipe(from:to:edge:progress:frames:format:time:)` beside `renderDissolve` — not a generalized `renderTransition(kind:)`.** A generalized method would need a renderer-facing kind enum carrying the edge — re-encoding `Transition` one level down and pushing a switch into every conformer; per-kind methods keep the `Compositor` the one place that maps a `Transition` to a renderer path and keep each kind's pixel work an explicit, separately testable method. `ShotRenderer` is a `TingraComposition` protocol, not the `TingraPlugInKit` stability contract, with exactly two conformers in-tree (`CoreImageShotRenderer` and the tests' mock) — so the new requirement ships with **no default implementation**, deliberately: a fallback (a wipe silently rendering as a dissolve, say) would mask a conformer that forgot the new kind, and the compiler error is the right tool at this seam.
  - **The Core Image wipe is soft-edged: a `CILinearGradient` mask driving `CIBlendWithMask`, feather fixed at 5% of the sweep span.** Both layer trees render through the existing per-shot compositing path (the dissolve's structure), then blend behind a grayscale gradient mask swept across the frame — built-in filters, still no custom shader. A hard edge would crawl as an aliased line at 30 fps; the narrow feather blends the moving boundary instead. The mask's blend band sits fully off-frame at progress 0 and fully past the far edge at progress 1, so the endpoints render the outgoing and incoming shot *exactly* — the renderer tests pin both, plus the mid-wipe geometry per edge (including the top/bottom Y-flip from operator terms into Core Image's bottom-left origin). An adjustable softness is a plausible later parameter of the wipe itself; this iteration keeps the contract to edge and duration. The output path is unchanged — the shared render tail vends the same IOSurface-backed 32BGRA program buffer, tagged BT.709 (no untagged buffer leaves the renderer).
  - **The switcher's boolean Dissolve checkbox becomes a segmented Transition picker (Cut ∣ Dissolve ∣ Wipe) with an Edge pop-up shown only while Wipe is selected.** A checkbox cannot name three kinds — the compelling reason the boolean `useDissolveTransition` is replaced by `takeTransitionKind` (+ `wipeEdge`) rather than extended. Segmented control per the HIG: a small, fixed set of mutually exclusive choices whose state must be visible and one click away on a *live* switcher (a pop-up would cost a click and hide the current kind); the wipe's edge is the secondary, set-and-forget choice, so it takes the standard pop-up beside it rather than a second segmented row competing with the primary control. Both remain per-take session state — transitions are still not persisted in the shot or preset (per-shot default transitions stayed a deferred iteration, since landed — see "Per-shot default transitions"), so **no document version bump, the format stays v2**. Each picker reports its `tap` event (`transition.picker`/`wipeEdge.picker`) in its `onChange`, where the user action executes — the pattern the checkbox's `transition.toggle` established; new strings localized `de`/`es` (the broadcast terms: Wischblende/Cortinilla).
  - **Taking with a wipe reports the same `program.take` event, its `transition` param extended with `"wipe"`.** The param is the kind discriminator (`Transition.eventName`) and stays exactly that — a small, stable vocabulary the CLI/JSON contract can key off; the edge joins the take no more than the dissolve's duration ever did (kind-specific detail stays out of the control-plane event, additively recoverable later if a consumer needs it).

**Per-shot default transitions (roadmap step 7, eleventh iteration).** The deferral recorded since the shot-management iteration lands: a shot carries the transition it is taken with, so the operator sets "this title always dissolves in" once instead of re-picking it per take. `Transition` was built for exactly this — a plain `Codable` value type "so it round-trips exactly wherever it is later persisted" (the 2026-07-08 record) — and with cut, dissolve, and wipe all landed, the vocabulary is complete enough to persist. Custom shader based transitions (since landed — see "Custom-shader transitions") and SRT/multiple destinations (step 8) remain deferred. Five decisions, recorded here (2026-07-18):
  - **The default lives on the `Shot` — an optional `defaultTransition: Transition?` — not the `Preset`.** The 2026-07-08 record left "a per-preset or per-shot default transition" open; per-shot wins because the preference is about *arriving at a shot*, not about the preset as a whole: a title shot wants to dissolve in while the interview beside it cuts, and one preset-wide default cannot say that (OBS's per-scene transition override is the same conclusion). A per-preset default also adds a second resolution tier (shot → preset → session) for little gain — a preset-wide house style is one setting repeated across its few shots, while the reverse (per-shot expression under a per-preset field) is unrepresentable. The field is optional and forgiving on decode: absent = no default = today's behavior, encoded only when set (the `Project.destination` rule), stable camelCase `defaultTransition` key, round-trip verified for every kind.
  - **No version bump — and a new standing rule (Larry, 2026-07-18): pre-release, the document format is simply version 1.** Nothing has shipped; the only readers of a `.tingraproject` are developer builds, so there are no older documents in the field to migrate and version churn buys nothing — the earlier v1→v2 `destination` bump is retired the same way (the format is v1 *with* the destination and the per-shot default; `Project.currentVersion` returns to 1). The versioning *machinery* stays exactly as built — the required `version` key, decode-newer-throws, forgiving optional decode — because it is the mechanism that matters once documents exist in the field: **version 2 happens the first time the format changes after the first release**, and migrations begin there. Until then the format grows freely within v1; a stale developer file written under the interim v2/v3 numbering decode-throws and is set aside/reseeded by the established unreadable-file path — the accepted, dev-only cost of carrying no migration debt.
  - **Resolution happens in the app (`EngineModel`), not the compositor — `take(shotID:transition:)` is untouched.** The 2026-07-08 contract stands: the compositor takes exactly the transition it is passed, because nothing about *how* a shot is taken belongs to the engine's switching primitive — the default is data on the shot, and resolving it against an operator's override reads UI session state (`takeTransitionKind`), which the compositor must not know about. `EngineModel.take(_:)` resolves the taken shot's `defaultTransition ?? .cut` when the picker is on Default, or the explicit kind otherwise, and passes the result down; `program.take` keeps reporting the **resolved** kind via `transition.eventName` — no new event shape, and the CLI/MCP callers keep the same primitive to build their own resolution on later.
  - **The switcher's segmented picker gains a leading Default segment (Default ∣ Cut ∣ Dissolve ∣ Wipe) — the initial selection — and the shot's default is set from a Default Transition submenu in its context menu.** Default-as-a-segment makes the resolution rule visible and one click away on the live switcher (the HIG reasoning that made it segmented), and making it the *initial* session state is what makes per-shot defaults effective out of the box — a shot with no default still cuts, so nothing changes until the operator asks for it; Cut/Dissolve/Wipe become explicit overrides (the Edge pop-up still rides the explicit Wipe only). Setting a default is shot *configuration*, not a take, so it lives with the other per-shot commands in the context menu (the macOS convention shot management established) as a checkmarked radio group: No Default, Cut, Dissolve, and a wipe per edge — all at the default durations the session picker itself takes with; per-default durations are a later editor concern, not a menu's.
  - **Setting a default is a document edit on the rename's exact line: through `Compositor.updateShot(_:)`, autosaved debounced, `tap`-event observability only.** The edited shot flows through the established in-place pool-edit path (so a mid-transition program is untouched — editing a shot is not taking it) and the debounced project-document autosave. It reports no new bus event: like a rename and unlike `shot.added`/`shot.removed`, the pool's membership is unchanged, and the context menu's `shotDefaultTransition.menu` `tap` event (kind + edge params) already records the discrete user action where it executes (EVENTS.md).

**Per-strip pan (roadmap step 7, twelfth iteration).** The next slot of the GLOSSARY.md **channel strip** lands after level and mute: **pan**, placing each strip in the stereo program mix (routing — since landed, see "Per-strip routing" — audio effect chains, and monitoring/meters — meters since landed, see "Per-strip meters" — remain deferred, as are custom-shader transitions — since landed, see "Custom-shader transitions" — and SRT/multiple destinations — step 8). The engine surface is deliberately small — a `pan` on `ChannelStrip` and an `AudioMixer.setPan(_:forInput:)` mirroring `setLevel` — with the substance in the pan law and the strip UI. Four decisions, recorded here (2026-07-18):
  - **Pan is a pair of per-program-channel gains applied at the mix tick, multiplying the strip's level gain.** `ChannelStrip` carries `pan: Double` — `-1` (hard left) through `0` (center, the default) to `1` (hard right), clamped at the mix (the negative-level rule's sibling); `setPan(_:forInput:)` applies from the next tick and ignores unknown ids, exactly `setLevel`'s contract, and like level it is a gesture-rate control reporting no engine events (EVENTS.md; the `updateShot` rule). Nothing else in the tick changes: the queues, the mute-drain rule, and the block's tick-time PTS are untouched.
  - **The pan law is equal-power (sine/cosine), normalized to unity at center — and one law serves both intake shapes: a mono strip pans, a stereo strip balances.** Equal-power is the console convention because it holds perceived loudness constant across the sweep (a mono source exactly, and centered stereo content too — the gains' squares sum to a constant); the *center-unity* normalization is chosen over the console's other convention (−3 dB center, unity at hard) because it makes a centered strip mix **byte-for-byte as it did before pan existed** — no behavior change to any session, and the pre-pan mixer tests staying green unchanged is the proof. The cost is the law's +3 dB (√2) on a hard-panned strip's remaining channel, which sits inside the mixer's recorded headroom posture — a float sum with no bus clamp, the same class of peak as a second unity strip, saturating at compression like any other sum. On a stereo strip the same two gains act as a **balance**: each source channel is scaled, never folded into the other (the DAW/broadcast stereo-channel convention), so hard-panning a stereo strip discards its far channel; channels past two are still dropped at intake. Implemented in the symmetric sine form (`sin` is exactly zero at zero where `cos(π/2)` is not), so a hard-panned strip's silent channel is exact silence, not a rounding residue.
  - **Pan stays session state — the levels-and-mutes calculus, unchanged by a third knob.** The strip list is still *derived from discovery*, not authored, so a device-keyed pan persisted before **routing** defines an authored channel would bake in the same shape routing will churn — pan joins levels and mutes in the persisted preset at the routing iteration ("The audio mixer", "Presets and shots"). When it does persist, it is an optional key within v1 under the pre-release rule ("Per-shot default transitions") — never a version bump. *(Since landed exactly so — see "Per-strip routing".)*
  - **The UI is a compact pan slider on each strip beside the level slider — broadcast `L`/`R` value labels, centered default, double-click to recenter.** A slider is the HIG control for a continuous bounded value, and the L/R value labels name the axis without a caption; double-click-to-recenter is the macOS convention for a slider with a meaningful default (the reset a centered detent would otherwise provide). Pan edits apply live, tick by tick, and report no engine events — the level slider's exact line — so observability is the drag-end `mixerPan.slider` `tap` (id plus where the pan landed) and, because a reset is a discrete action rather than a drag, the double-click's own `mixerPan.reset` `tap`. New strings localized `de`/`es` (Panorama/Panorámica; the `L`/`R` letters localize to `I`/`D` in Spanish).

**Per-strip meters (roadmap step 7, thirteenth iteration).** The monitoring half of the GLOSSARY.md **channel strip** lands: a **meter** per strip, showing the operator what each input is delivering (routing — since landed as exactly that persistence iteration, see "Per-strip routing" — and audio effect chains — the strip's remaining slots — stay later; custom-shader transitions — since landed, see "Custom-shader transitions" — and SRT/multiple destinations — step 8 — also remain). Display only: no control surface, no persisted state (nothing to persist for a meter), and no new engine events — `mixer.started`/`mixer.stopped` stay the mixer's only lifecycle events. Four decisions, recorded here (2026-07-18):
  - **Meter readings ride a dedicated single-consumer stream off the mix tick — never the event bus, and never a UI poll.** EVENTS.md's control-plane rule makes the transport decision: one reading per block (~47/s at the default format) is per-block data, categorically what the bus must never carry; and the engine's no-polling rule rules out the UI asking. The seam is `AudioMixer.meterReadings()` — an `AsyncStream<MeterBlock>` yielding one block per mix tick, stamped with the tick's master-clock time (tick-paced by construction, CLOCK.md), carrying a strip-keyed `MeterReading` for every live channel (a strip with nothing queued reads the floor, never a gap) — under `programAudio()`'s exact one-consumer, replace-on-recall contract. The measurement is a byproduct of the walk the tick already makes over every channel's samples — never a second pass — and is taken only while a consumer is attached, so an unwatched meter costs nothing and the streamed program path is byte-for-byte unchanged.
  - **Metering is pre-fader: the reading measures the strip's intake — after normalization, before level, pan, and mute.** The strip meter answers the operator's diagnostic question — *what is this input delivering* — so it holds steady while the operator rides the fader (the console input-strip convention; post-fader metering belongs to the master bus, a later monitoring slice). It also composes with the future monitoring path: engine-side, a muted strip still meters its live signal. In the app a muted strip's meter rests at the floor anyway — the mute-stops-device policy means no samples arrive — but that is app policy, not meter semantics; a monitoring path that keeps a muted device capturing gets a live meter for free. An underrunning strip is measured over exactly the samples it delivered (never the silence that pads the block), and readings are linear magnitudes — the dBFS scale is presentation, not signal truth.
  - **Each reading carries both peak and RMS — one pass yields both, and the meter needs both.** Peak is the headroom signal (how close the block came to full scale), RMS the loudness signal; delivering one and wanting the other later would mean a seam change, where both are two floats off samples already in hand (max-across-channels for each, so a hard-panned stereo strip meters its hot channel). Ballistics are deliberately *not* in the engine: readings stay raw per-block truth — deterministic under the synthetic clock (known samples → known readings; silence → the floor) — because decay is a display convention that belongs to the display.
  - **The UI is a custom capsule drawn at display cadence off a shared relay — the `ProgramFrameRelay` pattern, not observation, and not `Gauge`.** A `MeterRelay` holds the latest tick's readings; the model's one meter drain writes it and each strip's `StripMeter` samples it inside a `TimelineView(.animation)` — readings never pass through SwiftUI observation, so the reading cadence causes no observation-driven layout churn (the exact reasoning that put the video preview behind `ProgramFrameRelay`, CLOCK.md's preview-sampling rule applied to audio display). SwiftUI's `Gauge` is rejected because its styles cannot express a broadcast meter — fixed zone colors, a separate peak marker, draw-time ballistics. The capsule renders the RMS bar over broadcast green/yellow/red zones (green through −20 dBFS, yellow through −6, red above, on a −60…0 dBFS scale) with a decayed peak line; ballistics are instant attack (the operator must never miss a hot block) with a 20 dB/s decay — between IEC PPM's ~12 dB/s and fast digital meters' ~40 dB/s, brisk enough to track speech, slow enough to read. The meter sits between the strip's name and its level slider (the console strip's meter-beside-fader arrangement) and reports no `tap`s — there is nothing to click. New string localized `de`/`es` (Pegelanzeige / Medidor de nivel).

**Per-strip routing: the persisted mix (roadmap step 7, fourteenth iteration).** The **routing** slot of the GLOSSARY.md **channel strip** lands — and with it the debt three records carried: strip settings (level, pan, mute) finally **persist**, retiring the "joins the persisted preset when routing lands" deferral recorded since the mixer landed ("The audio mixer", "Per-strip pan", "Per-strip meters"). Routing defines the **authored channel** the discovery-derived strip list has been waiting for; audio effect chains — the strip's last slot — stay later, as do custom-shader transitions (since landed — see "Custom-shader transitions") and SRT/multiple destinations (step 8). Five decisions, recorded here (2026-07-19):
  - **Routing v1: the program mix is the only bus, so a channel's routing *is* its membership in the preset's authored channel list — no new engine surface.** With one bus, GLOSSARY.md's routing ("where a channel strip's signal goes") collapses to "does this strip feed the program mix," which membership plus mute already express — so `AudioMixer` and `ChannelStrip` gain no API this iteration: the deliverable is persistence, the authored set replacing "derived from discovery" as the strips' source of truth. Sends and additional buses are deliberately later — they would be the first *second* destination for a strip's signal, and nothing before the audio-preview monitoring slice needs one.
  - **The document type is `AudioChannel` — per-preset, as always slated — living in `TingraComposition` beside `Preset`.** A plain `Codable` value on the project/scripting contract (stable camelCase `input`/`name`/`level`/`pan`/`isMuted` keys; `input` required, everything else decoding forgivingly to the strip defaults); `Preset` gains an optional `audioChannels: [AudioChannel]?` — encoded only when authored, **no version bump** (the pre-release rule): `nil` means no authored audio, so every pre-routing document decodes unchanged, while an empty array is authored-none. Per-preset survives contact with the strip-follows-device reality because a preset's audio configuration is the console's *scene snapshot* — two presets hold their own settings for the same device UID, which is exactly what "switch between while live" promises (GLOSSARY.md, "Preset"). The type sits beside the document it serializes into (the `ProjectDestination` precedent — document types live together), not in `TingraAudio`, whose `ChannelStrip` stays the live, deliberately non-`Codable` strip carrying the running input.
  - **The persisted identity is the device UID the `InputID` already carries; an absent device's channel stays a dormant strip.** A microphone's `InputID` raw value is its `AVCaptureDevice.uniqueID` — stable across reconnection and relaunch — so the channel persists the id the strip already has, plus a cached `name` so the strip stays labeled while its device is away. A channel whose device is not discovered at launch keeps its place on the panel — settings editable, metering at the floor, contributing silence until the device returns (marked not connected, `mic.slash`) — the layer-bound-to-an-undiscovered-input semantic ("Project save/load"), one service over.
  - **The merge policy: authored channels first, in document order; unauthored discovered devices append muted at unity, centered; the seed policy is the no-document fallback.** The panel order *is* the persisted array order (switcher order is array order — "Shot and preset reordering", one surface over). A device the preset never authored is never surprise-live: it appends muted (the honest-microphone posture — nothing captures until the operator asks), where the seed's "first microphone unmuted" applies only when *nothing* is authored (`audioChannels` nil — a fresh project or a pre-routing document), preserving `MixerStrip.seed` as exactly the fallback it was.
  - **Strip edits autosave like shot edits, the session strips sync into the active preset, and a preset switch adopts the incoming preset's authored audio.** Level, pan, and mute changes schedule the same debounced autosave as layer edits (a drag coalesces into one write — the `updateShot` rule; still no engine events, the `tap`s carry observability unchanged). The session mix *is* the active preset's audio configuration — synced back before every save, switch, and duplicate (the `syncActivePreset` rule extended), so the first save after routing lands authors a pre-routing document's active preset by use: the "grows within v1" path, no migration. Switching presets applies the target's channels from the next mix tick — a control change, never an interruption (the seamless rule), with the multiple-presets record's "strips carry across untouched" surviving as exactly the no-authored-audio case — and removing the active preset adopts the adjacent preset's audio the same way.

**Custom-shader transitions (roadmap step 7, fifteenth iteration).** The fourth and last GLOSSARY.md transition kind lands — **shader**, a custom Metal-shader reveal — completing the transitions story cut and dissolve opened and closing the deferral every record since 2026-07-08 carried. This is also where **hand-written Metal finally arrives** in the repo, exactly where the composition-renderer decision sequenced it ("raw Metal shaders arrive only where custom work or performance demands them, at a later effects/transitions step"): the wipe record deferred "diagonals, irises, and patterns" as shapes rather than edges, and those shapes are this iteration's menu. The tick-counted timing spine is untouched — a shader transition is the dissolve's `PendingTransition` with a different renderer path — so the new work is one `Transition` case, one `ShotRenderer` requirement, the Metal kernels behind one Core Image implementation, and the switcher UI. Audio effect chains and per-layer video effects (the step-7 exit criteria's effect subsystem) and SRT/multiple destinations (step 8) remain deferred. Five decisions, recorded here (2026-07-19):
  - **`Transition` gains `.shader(name:duration:)` with a `TransitionShader` menu (`iris`/`diagonal`/`blinds`) — an additive change to the project/scripting contract.** The Codable form stays stable camelCase and round-trip exact: `kind: "shader"` plus a `shader` key (the `TransitionShader` raw value — the kind-specific aspect, exactly the wipe's `edge`) beside `durationSeconds`; both optional on decode (the iris, the half-second default — the forgiving-decode rule), while an **unknown shader name throws** like an unknown `kind` or `edge` — a document naming a shader this build does not carry must not silently take with a different look. The menu's three entries are the shapes the wipe's straight edges cannot draw; growing the menu is an additive case (pre-release, within v1), and the raw value doubles as the Metal function name the renderer compiles, so the contract and the kernel can never drift apart.
  - **The kernels are hand-written stitchable Metal, runtime-compiled from compiled-in source — no Metal-library build machinery.** `CIKernel.kernels(withMetalString:)` (macOS 14+, inside the macOS 15 floor — no availability guard) compiles the first-party source once, lazily, inside the task-confined renderer; a session that never takes a shader transition never compiles. SPM's resource pipeline cannot pass the Core Image kernel toolchain flags a prebuilt `.ci.metallib` needs, so runtime compilation of vendored source is both the clean build story and — verified before landing — renders correctly under the **software `CIContext`**, keeping the pixel tests deterministic and GPU-free (the endpoint tests double as compile checks: a kernel that failed to build would degrade visibly). All three kernels share one signature (both shots' samples, progress, the program size, the destination) and the wipe's sweep rule — a 5%-feathered band running from fully off its span at progress 0 to fully past it at 1, so the endpoints are exact — and coordinates flip from Core Image's bottom-left origin into the operator's top-left screen terms (the `placedImage` flip: the diagonal opens from the screen's top-left, blinds reveal downward).
  - **Security posture: v1 runs first-party shader source only — third-party shader loading is a deliberate non-goal, deferred with its own boundary to design.** The compiler's input is a fixed set of string constants compiled into the binary; no path reads shader code from a project document, a file, or user input — a document can only *name* a `TransitionShader` menu entry, and an unknown name throws at decode. Executing user-supplied GPU code is a real security boundary, not a missing feature: a hostile kernel runs with the engine process's GPU access against surfaces the process holds, and a malformed one can hang the GPU — so a third-party shader seam is plug-in-era work that gets its own recorded posture (what is accepted, how it is validated and isolated) *before* any loader lands, the same decide-then-build rule the effect seam follows.
  - **The seam grows per-kind, and a kernel that cannot run degrades to the incoming shot — never a crash, never a frozen program.** `renderShader(from:to:shader:progress:frames:format:time:)` joins `renderWipe` with no default implementation (the wipe record's reasoning: a silent fallback would mask a conformer that forgot the kind; the seam is package-internal, the compiler error is the right tool), and the compositor's `BlendKind` maps the case — still the only place a `Transition` picks a renderer path. If the first-party kernel fails to compile or apply — an environment defect, never expected — the renderer returns the *incoming* shot's plain render (visually, the transition becomes a cut): returning `nil` would make the compositor skip every tick of the transition and blank the program for its duration, where the never-crash rule wants a live program and an honest, visible degradation.
  - **The switcher's picker gains a Shader segment with a shader pop-up, and the Default Transition submenu gains the three shaders — the wipe's exact UI pattern.** The segmented picker stays the primary, always-visible kind choice (Default ∣ Cut ∣ Dissolve ∣ Wipe ∣ Shader); the shader is the secondary set-and-forget choice, so it takes the pop-up slot the wipe's Edge pop-up established, shown only while Shader is selected, reporting `shaderName.picker` `tap`s in its `onChange` (and the context menu's `shotDefaultTransition.menu` `tap` gains a `shader` param beside `edge`). A shader default persists through `Shot.defaultTransition` unchanged — `Transition` was built to round-trip wherever it lands, so the document format needs nothing (no version bump; the pre-release rule). `program.take` keeps reporting the kind discriminator alone (`"shader"`), the wipe record's small-stable-vocabulary rule. New strings localized `de`/`es` (the broadcast terms: Jalousie/Persiana for blinds; Shader, Iris, and Diagonal are the loanwords both languages use).

**The effect seam (roadmap step 7, decision record — no code yet).** The last step-7 exit criterion is the effect subsystem: **audio effect chains** (the final GLOSSARY.md channel-strip slot) and **per-layer video effects** (the "Effect" iteration deferred since the layer-tree editor). Effects are plug-ins (CLAUDE.md — the archetypal *third-party* plug-in, in fact), so the seam lands in `TingraPlugInKit` under the plug-in API stability contract — which is exactly why its shape is decided and recorded **before any code**: a half-designed stability contract is worse than an unfinished step, because it is expensive to walk back after 1.0. Decided 2026-07-19, **flagged for Larry's veto before any conformance code lands**; the build follows as two iterations whose own records carry the per-iteration details (chain placement in the signal order, parameter UI, the first-party effect menu):
  - **One effect seam, two media protocols — `AudioEffect` and `VideoEffect`, never one protocol straddling both.** The recording-seam precedent controls: when streaming and recording shared the output concept, the repo chose a narrower separate protocol through the same registering seam over one stretched protocol, and the same forces apply here — an audio effect processes the mixer's native currency (blocks of deinterleaved float32 at the mix rate, at the mix tick), a video effect the renderer's (GPU-resident images at the program tick), and one protocol covering both would need either default no-op requirements (a conformer could silently be neither — the exact failure the default-implementation rule exists to prevent, turned inside out) or an associated-type media parameter that pushes generics through every registry, chain, and persisted document. Two protocols keep each side's processing signature native and separately evolvable inside the SemVer rules.
  - **The seam is shared everywhere the media type does not appear**: one registration surface (`EffectRegistering`, beside `InputRegistering`/`OutputRegistering`/`ToolRegistering` in the protocol package), one identity model (a stable string effect identifier plus display name), one provider/instance split (the plug-in registers an effect *provider*; the engine instantiates per chain slot — the `StreamingServiceProvider`/`StreamingService` pattern), and one persisted parameter shape (a `Codable`, stable-camelCase payload on the project/scripting contract). GLOSSARY.md's effect/filter vocabulary (an effect may chain several filters) applies to both media identically.
  - **Chains persist as ordered lists on the documents that own them — an optional `effects` key on the authored `AudioChannel` and on the `Layer`, within v1** (the pre-release rule, no version bump): absent means no chain, so every existing document decodes unchanged. Order is signal order — the chain *is* its array, the switcher-order-is-array-order rule applied to processing.
  - **Video currency is Core Image (`CIImage → CIImage`).** It is the only GPU-resident currency that composes lazily — the renderer fuses a whole chain into one render pass, where per-effect `CVPixelBuffer` hand-offs would materialize an intermediate buffer per effect per layer per tick and defeat the GPU-resident rule. CoreImage enters the protocol package the way CoreVideo/CoreMedia already did (`CapturedFrame` wraps `CVPixelBuffer`): an Apple media framework as seam currency, not a third-party dependency. A custom Metal kernel inside an effect is the effect plug-in's own business — for third parties, running their *code* is the existing external-bundle-loader boundary, not a new one; the shader-transition posture (first-party compiled-in source only, until a loader's security posture is recorded) stands unchanged.
  - **First-party effects land in a new `TingraEffectPlugIns` package** (the established plug-in-package pattern: protocol-seam dependencies only), as **two iterations against the shared seam — audio effect chains first** (the channel strip is one slot from complete, and its persistence path — the authored `AudioChannel` — just landed), **then per-layer video effects**. Which concrete effects ship first-party (GLOSSARY.md names color adjustment, blur, keying, stylization for video; gain/EQ/compression are the audio staples) is each iteration's decision, not this record's.

### Compression
- Compression happens at the sinks; Tingra does not own a standalone compression stage. For streaming, HaishinKit compresses internally (hardware H.264/HEVC via VideoToolbox) as part of output, with settings (bitrate, keyframe interval, profile) driven by the selected destination through the stream configuration. For recording, `AVAssetWriter` performs its own hardware encode.
- Both sinks accept GPU backed pixel buffers directly, so the program frame feeds in without a CPU copy.

### Audio
- **AVAudioEngine / Core Audio** for capturing audio inputs (microphones, application audio, media); audio effects and taps are plug-ins. The mixing stage is the engine's `AudioMixer` (`packages/TingraAudio`, roadmap step 7): a clock-paced software mix loop over one channel strip per input, with `AVAudioConverter` normalizing each strip's audio once at channel intake — see "The audio mixer" above. Produces the mixed program audio fed to streaming output (and recording, when the app records), plus per-strip meter readings off the same mix tick (see "Per-strip meters"). Strip settings persist per preset as authored `AudioChannel`s routed to the program mix — v1's only bus (see "Per-strip routing").

### Output
- **HaishinKit** (BSD 3-Clause) handles RTMP and SRT output: the connection handling, retry logic, and muxing that Apple provides no framework for. WHIP/WHEP ships as alpha in `RTCHaishinKit`, giving a path to WebRTC output later.
- **Local recording** uses `AVAssetWriter` to write `.mov`/`.mp4` directly, independent of streaming output — the first-party recording plug-in (`TingraRecordingPlugIns`, roadmap step 5), behind the `RecordingService` seam and registered through the same `OutputRegistering` seam as streaming (see "The output registration seam" above). It is fed the same program media, so both sinks record the identical program; the recording carries the same BT.709 delivery color tags as the stream. Unlike HaishinKit (which detects tracks from the buffers it is appended), `AVAssetWriter` must declare its tracks up front, so the program's track topology travels in `StreamConfiguration` (`includesVideo`/`includesAudio`).

## How HaishinKit is incorporated (minimal adoption approach)

HaishinKit is capable of doing capture, video mixing, and output on its own. Tingra deliberately uses **only its output layer**, for three reasons: it preserves Tingra's distinctive native capture/composition story, it avoids inheriting HaishinKit's architecture across the whole app, and it isolates the dependency to a single, well defined boundary.

**The clean seam:** Tingra's Metal compositor produces the program frame; from that point, HaishinKit's stream objects (`RTMPStream` / `SRTStream` in HaishinKit 2.x) are responsible for compression and output to RTMP/SRT. In practice this means appending Tingra's program video and mixed audio to the stream directly, rather than attaching HaishinKit's own `MediaMixer` capture to the camera or screen.

Concretely:
- Tingra owns `AVCaptureSession` / `SCStream` and the Metal compositor.
- Tingra appends program video and mixed audio buffers to HaishinKit streams; HaishinKit's `MediaMixer` and device capture are not used.
- All compression for streaming, RTMP/SRT connection setup, reconnection, and muxing is delegated to HaishinKit.
- A thin `StreamingService` protocol wraps HaishinKit so the rest of Tingra never imports it directly. This keeps the dependency swappable and the seam explicit; the HaishinKit backed implementation is itself an output plug-in.

**The output registration seam** (decided 2026-07-04, mirroring the input seam): a `StreamingService` is per-session state, so what an output plug-in registers is a **`StreamingServiceProvider`** — a factory declaring the destination URL schemes it serves (`rtmp`/`rtmps` for the HaishinKit RTMP provider) and creating a configured `StreamingService` per stream. The provider and the `OutputRegistering` seam protocol live in the plug-in protocol package; the host's `OutputRegistry` conforms and arrives through `PlugInContext.outputs`, exactly as `InputRegistry` arrives through `PlugInContext.inputs`. The engine resolves a destination to a provider by URL scheme; one provider per scheme, duplicates rejected at registration. Both front ends resolve through this same seam: the CLI's `stream` command and the phase-3 app both activate `HaishinKitOutputPlugIn` and ask `OutputRegistry.provider(forScheme:)` for the service — the app feeds it the compositor's program frames through `StreamSession`'s program source (see "Streaming the program").

**Recording joins the same seam as a parallel provider kind** (decided 2026-07-05, roadmap step 5). Recording writes the program to a local file (`AVAssetWriter`), a compression sink parallel to streaming (GLOSSARY.md, "Output": to destinations *or* to a recording). It is registered through the same `OutputRegistering` seam and held by the same `OutputRegistry` — one registry, two provider kinds — but through a **narrower `RecordingService` / `RecordingServiceProvider` pair rather than reusing `StreamingService`**. The reasons the narrower protocol wins over a "file"-scheme `StreamingServiceProvider`: a recording has no `Destination` (a file carries no stream key), no `connectionLost`/reconnect (a file write is terminal, not recoverable), and is resolved by the `--record` path's **file extension** (`mov`/`mp4`), not by a URL scheme — because recording runs *alongside* streaming (either or both active), not as the single scheme-resolved destination. Forcing recording into `StreamingService` would carry a meaningless secret and drag the reconnect machinery onto a sink with no connection. So the host resolves a recording target to a `RecordingServiceProvider` by extension (one provider per extension), and `StreamSession` drives the recording sink from the same rebased program media it feeds the stream — the recording keeps writing across a reconnect gap and is always finalized on teardown, however the session ends. A mid-recording write failure is reported as a `recordingFailed` error event and stops the recording, but never ends the stream (recording is independent); a recording that cannot even open fails the run before anything streams.

```swift
// Sketch of the boundary — not final API.
protocol StreamingService {
    func start(to destination: Destination) async throws
    func send(video frame: CVPixelBuffer, at time: CMTime)
    func send(audio buffer: CMSampleBuffer)
    func stop() async
}

// HaishinKit-backed implementation lives behind this protocol,
// so no other module in Tingra depends on HaishinKit directly.
struct HaishinKitStreamingService: StreamingService { /* ... */ }
```

### Dependency notes / caveats
- **Project home:** the library lives at the HaishinKit GitHub org (`HaishinKit/HaishinKit.swift`, formerly under shogo4405). Latest release at review time: 2.2.5 (March 2026), Swift 6, strict concurrency compliant.
- **License:** HaishinKit is BSD 3-Clause (verified), permissive and compatible with Tingra's MIT license.
- **Modules (2.x):** core `HaishinKit` plus `RTMPHaishinKit`, `SRTHaishinKit`, `RTCHaishinKit` (WHIP/WHEP, alpha), and `MoQTHaishinKit`. Pull in only what Tingra ships: core + RTMP first, SRT when added.
- **Binary targets (resolved):** the only prebuilt piece is the libsrt xcframework (1.5.4) inside `SRTHaishinKit`; all Swift code is source. An RTMP only build is fully source. Adopting SRT means shipping the prebuilt libsrt, or building libsrt from source ourselves. If that ever becomes unacceptable, the `StreamingService` seam lets us swap implementations without touching the rest of the app.
- **Integration:** Swift Package Manager.
- **Platform floor:** ScreenCaptureKit needs macOS 12.3+, ScreenCaptureKit system audio capture needs 13.0+, and HaishinKit 2.x needs macOS 12.0+ and a Swift 6 toolchain. Deployment target for Tingra: **macOS 15, Apple Silicon (arm64) only** — every Apple Silicon Mac runs macOS 15, so the floor excludes no hardware. Development toolchain floor: Xcode 26.6 / Swift 6.3.3, enforced by CI on GitHub Actions (see CLAUDE.md, "Toolchain & CI").

## Repository structure

The repo is a monorepo (multi package workspace) with two top-level folders. **`apps/`** holds the runnable products: `tingra-cli` (see CLI.md), `ingest-simulator` (the local RTMP/SRT test server, see SIMULATOR.md), and the assembled `tingra` app (phase 3, scaffolded at roadmap step 6). **`packages/`** holds the engine libraries the apps build on. The package names are finalized (reviewed 2026-07-03): **`TingraHost`** (the host/core package), **`TingraPlugInKit`** (the plug-in protocol package, importable by third parties without the engine), and **`TingraEventBus`** (the event bus both build on, see EVENTS.md); first party feature plug-ins land as their own packages alongside them: **`TingraCapturePlugIns`** (camera, microphone, and display), **`TingraGeneratorPlugIns`** (bars and tone, the permanent CI test surface), **`TingraOutputPlugIns`** (the HaishinKit backed streaming output — the only package that imports HaishinKit), and **`TingraRecordingPlugIns`** (the `AVAssetWriter` backed local recording; a separate package so `TingraOutputPlugIns` keeps its defining property — the only HaishinKit importer — and recording pulls in neither HaishinKit nor Logboard). Composition lives in **`TingraComposition`** (the tick-paced Metal/Core Image compositor, the layer tree, and the program format; roadmap step 6) — a host-side engine library depending only on `TingraPlugInKit` and `TingraEventBus`, not a plug-in and not folded into the minimal `TingraHost`. The audio mixer lives in **`TingraAudio`** (the clock-paced `AudioMixer` and its channel strips; roadmap step 7) — a sibling engine library on the same protocol-only dependency rule (see "The audio mixer"). The MCP/Control service lives in **`TingraMCP`** (the `serve` daemon, the hand-rolled MCP JSON-RPC layer, the `mcp` stdio↔socket proxy, and the first-party control tools); it depends on `TingraHost` because the daemon owns the engine, while the `ToolRegistering` tool seam itself lives in `TingraPlugInKit` alongside the input and output seams. UI packages arrive under `packages/` in phase 2.

## UI layer
- **SwiftUI + AppKit**, with Metal preview content hosted in an `MTKView` (or a `CAMetalLayer` bridged into the view hierarchy).
- The interface exposes presets and shots, inputs, the audio mixer, program/preview monitoring, recording controls, and destination configuration.
- **Scaffolded at roadmap step 6** (`apps/tingra`): a SwiftUI `@main` app whose one `@Observable @MainActor` engine model boots the host and activates the same capture and generator plug-ins through the same `PlugInContext` the CLI uses — the app takes shape around the proven engine, not a fork of its pipeline. ## Summary

Tingra owns the parts that make it native and distinctive (capture on ScreenCaptureKit/AVFoundation, composition on Metal, audio on AVAudioEngine) and delegates the standardized, undifferentiated compress and send work to HaishinKit behind a thin `StreamingService` protocol. Everything a user would call a feature is a plug-in against a minimal host, first party and third party on the same protocol. The entire media path stays GPU resident from capture to the compression sinks, and the HaishinKit dependency is isolated to a single swappable seam, keeping the project MIT clean and future proof.
